Recommended deployment stages
1. Local development

Use this first:

Docker Compose
- app
- postgres
- redis
- optional worker
- optional preview service

This should remain the easiest way to run the product.

2. Cloud dev / testing

Best Azure reference for early testing:

Azure Container Apps
Azure PostgreSQL
Azure Blob Storage
Azure Redis

This is the sweet spot for early development because it is much simpler than AKS.

3. Production reference

For larger deployments:

AKS
Managed PostgreSQL
Blob Storage
Redis
Separate worker pools


Core runtime components
Web/API app

This is the main Next.js app.

Responsibilities:

admin UI

viewer UI

authenticated API

permission checks via PermissionEngine

session handling

room/document operations

viewer rendering and watermark application

Keep this stateless so multiple app instances can scale horizontally.

Worker tier

Split async work away from the web tier.

Suggested worker types:

general-worker
preview-worker
scan-worker
report-worker

Why separate them:

preview generation is CPU-heavy

virus scanning can be slow

reports and exports can be long-running

emails and webhooks should not compete with preview jobs

Use Redis for:

Recommended request flow
Upload flow

User uploads file
→ app stores original in object storage
→ DB record created
→ upload event emitted
→ job queued
→ scan worker scans
→ preview worker converts
→ text extracted
→ thumbnails generated
→ search index updated
→ document becomes viewable

Viewer flow

Viewer opens room
→ session validated
→ PermissionEngine checks access
→ NDA / password / link policy checked
→ preview served
→ watermark applied at render time
→ activity event emitted
→ analytics updated asynchronously

Security architecture
Minimum secure setup

For MVP/V1, I’d recommend:

HTTPS everywhere
Private storage container/bucket
Signed temporary access URLs only internally
Server-side permission checks for every action
Redis-backed rate limits
Virus scanning before public availability
Immutable audit events
Backups enabled from day one

Design rules I strongly recommend
1. Keep the app stateless

No local-only required state in app containers.

2. Never do heavy processing in request path

All conversion, scanning, exports, OCR, and report generation should be queued.

3. Keep storage private

Only serve documents through authenticated application logic or tightly scoped signed access.

4. Separate originals from derivatives

Use different storage paths or containers for:

originals

previews

thumbnails

exports

5. Make the worker model explicit

Do not let “background jobs” remain abstract. Define worker roles early.
