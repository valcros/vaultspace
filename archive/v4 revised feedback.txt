1. Architectural Decisions That Are Excellent
EventBus as a core primitive

This is the single best architectural choice in the document.

The system now explicitly states:

All state changes emit events. Database-backed event log.

Event schema defined:

event_id
event_type
timestamp
actor_id
actor_type
organization_id
room_id
document_id
metadata_json
ip_address
user_agent

This allows:

audit logs

analytics

webhooks

notifications

compliance exports

chain of custody reports

to all be built on the same data model.

That dramatically reduces complexity.

Suggestion

Add:

request_id
session_id

This allows grouping events across a single request.

Extremely useful for debugging and security forensics.

2. PermissionEngine is critical and correctly added

Adding F141 centralized permission evaluation was a major improvement.

Centralizing permissions prevents the classic issue where:

UI

API

background jobs

all evaluate permissions differently.

Your model:

canUserAccessDocument(user, document, action)

is the correct design.

Suggestion

Add permission explanation capability.

Example:

explainPermission(user, document, action)

Returns something like:

Allowed because:
- user belongs to group "Investors"
- group has read permission on folder
- document does not override ACL
- link policy allows view

Admins frequently ask:

"Why can this person see this document?"

This tool solves that.

3. Multi-tenant from day one is extremely important

This was a very smart decision.

Every database entity includes organization_id.

Many projects start single-tenant then regret it later.

Your approach:

organization_id on all entities

allows:

SaaS later

organization isolation

multi-room multi-org hosting

with zero migration.

Excellent decision.

Architecture Areas That Could Still Improve
1. Search architecture should be more explicit

Currently search is:

Postgres FTS (MVP)
Meilisearch (V1)
OpenSearch/Elastic (V2)

But search will eventually require:

separate service
separate indexing jobs
schema versioning

Recommendation:

Create a SearchIndex model separate from documents.

Example:

search_index
-------------
document_id
version_id
organization_id
text
metadata
vector_embedding

This allows search engines to be swapped without reprocessing documents.

2. Document object model should be clarified

Right now documents include:

version
hash
preview
metadata

But it would help to explicitly separate:

Document
DocumentVersion
FileBlob
PreviewAsset
ExtractedText

Typical structure:

Document
 ├ DocumentVersion
 │   ├ FileBlob
 │   ├ PreviewAssets
 │   ├ ExtractedText
 │   └ Hash

This prevents version history from becoming messy.

3. Event log storage strategy

The event bus writes to the database.

That’s good initially.

But event tables grow extremely fast.

You should plan for:

partitioning
retention policies
event archiving

Example approach:

events_2026_01
events_2026_02

Postgres partitioning helps.

4. Preview generation scaling

Preview generation can become the largest CPU consumer.

The system should support:

multiple preview workers
job concurrency control

Example:

preview-worker
scan-worker
email-worker
analytics-worker

Even if implemented later.

5. Encryption architecture

Feature:

EncryptionProvider
AES-256 file-level encryption

But encryption design should answer:

Where are keys stored?

Options:

env variable
config file
vault
cloud KMS

Recommended MVP:

AES256 key from env

Later:

external KMS adapters
